# Uinify v0.3 Theming And Slot Contract Design

## Summary

`v0.3` should polish theming and slot contracts so consumers can integrate `uinify` into real design systems with more confidence.

The current library already has theme tokens, slot hooks, and renderer overrides, but the stable contract boundaries are still not as clear or complete as they should be for long-term integration.

## Goal

After this workstream, consumers should know which styling hooks are safe to depend on, how to override visuals correctly, and how to integrate `uinify` with a design system without relying on brittle DOM knowledge.

## Scope

This workstream includes:

- stable slot contract clarification
- slot and state metadata consistency
- theme token coverage polish
- styling integration guidance

## Slot Contract Direction

`v0.3` should explicitly define which slots are stable.

Consumers should not need to infer styling hooks from implementation details or nested DOM structure.

This workstream should improve:

- stable slot naming
- stable state metadata on slots
- consistency across transcript, artifact, composer, attachment, and action regions

## Theme Token Direction

Theme token coverage should be extended or clarified where needed so major surfaces have a complete enough story.

Priority areas include:

- transcript regions
- composer regions
- attachment surfaces
- artifact container surfaces
- action surfaces

## Styling Guidance

The docs and contract should help consumers answer:

- when to use theme tokens
- when to use slot-specific class hooks
- when to replace renderers instead of styling the default ones

The goal is not more flexibility at any cost. The goal is clearer intended usage.

## Non-Goals

This workstream does not include:

- new visual language redesign
- runtime theme builder
- visual theme editor
- complex token engine expansion

## Testing Strategy

Testing should include:

- stable slot presence checks
- state metadata consistency
- override and slot coexistence tests
- token coverage smoke verification where practical

## Success Criteria

This workstream is successful when:

- consumers can rely on documented slot surfaces instead of DOM accidents
- styling override paths are clearer and less overlapping
- design-system integration becomes easier to reason about
- renderer overrides and slot contracts remain compatible

## Recommendation

Keep this focused on clarity and completeness, not breadth.

The point is to make integration safer, not to introduce a new styling platform.
