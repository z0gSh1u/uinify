# Uinify v0.3 Attachment Lifecycle Phase 2 Design

## Summary

`v0.3` should deepen the current attachment lifecycle so it can support more realistic product behavior.

`v0.2` established lifecycle states, composer integration, host-driven updates, and transcript rendering. `v0.3` should extend that work so attachments can better support validation, failure handling, cancellation intent, and clearer continuity between composer and transcript phases.

## Goal

After this workstream, attachment orchestration should feel realistic enough for product integration, not just representable in state.

## Scope

This workstream includes:

- validation and rejection contract
- clearer progress and failure UX
- cancel or interruption hook points
- stronger continuity between pre-send and message-bound attachment flows

## Validation And Rejection

`v0.3` should define a clearer contract for:

- file type validation
- size validation
- count limits
- rejected-attachment feedback

The boundary should remain:

- host decides policy
- library renders feedback and state

## Progress And Failure UX

The attachment UI should communicate not only state, but meaning.

That means clearer support for:

- why send is blocked
- why upload failed
- what retry or remove will do
- what stage the attachment is currently in

## Cancel And Interruption Hooks

The library should expose clear hook points for host-owned actions such as:

- cancel upload intent
- retry intent
- remove intent

The library should still avoid implementing actual upload transport.

## Composer To Transcript Continuity

The current split between pre-send and message-bound attachment contexts is correct, but it should become easier for hosts to keep those phases coherent.

`v0.3` should improve the contract so the transition from composer attachment to transcript attachment is more legible and less host-specific guesswork.

## Non-Goals

This workstream does not include:

- official upload client
- storage provider integrations
- signed URL workflow framework
- file manager UI

## Testing Strategy

Testing should include:

- validation and rejection behavior
- send gating and send-blocking reason behavior
- retry, remove, and cancel intent hooks
- composer and transcript continuity scenarios
- controlled host orchestration under complex state transitions

## Success Criteria

This workstream is successful when:

- hosts can implement realistic upload orchestration without fighting the UI layer
- validation and failure paths are as clear as success paths
- send gating behavior is explainable and testable
- attachment state continuity across composer and transcript feels intentional

## Recommendation

Keep this focused on orchestration contract and UX clarity, not network transport.

That is where `uinify` can provide real product value while preserving backend independence.
