# Uinify v0.4 Composable Agent UI Kit Design

## Summary

`v0.4` should move `uinify` from a backend-agnostic chat UI foundation into a layered, composable agent UI kit.

The release should focus on product-grade agent UI primitives without turning the library into an agent runtime, workflow engine, or backend framework. The core thesis remains React-first, backend-agnostic, host-controlled orchestration with opinionated defaults and stable extension seams.

The v0.4 theme is: **Composable Agent UI Kit**.

## Goals

After v0.4, consumers should be able to:

- Render agent execution as a sequence of lightweight steps.
- Represent tool calls as `step` parts with `category: "tool"` instead of a separate tool-call model.
- Render image parts through canonical stream events.
- Move image attachments from composer submit into transcript message parts.
- Use a unified composer command contract for slash commands, mentions, agents, MCP tools, skills, and custom commands.
- Understand the public API as layered primitives rather than a narrow entrypoint plus undocumented escape hatches.

## Non-Goals

v0.4 will not include:

- Automatic command execution.
- MCP client or server integration.
- Official OpenAI, Anthropic, OpenCode, AG-UI, or vendor adapter packages.
- Workflow DAGs, dependency graphs, or observability-style tracing.
- Rich preview registry for PDF, CSV, audio, or arbitrary files.
- Voice input.
- Backend upload services.
- A full application shell, session list, or SaaS features.

## Architecture

The architecture should remain layered.

### Core Layer

The `uinify` package root owns canonical UI state and events.

Stable core responsibilities:

- `createChatRuntime`
- `UiRuntimeState`
- `UiMessage`
- `UiMessagePart`
- `UiStreamEvent`
- adapter helpers such as `createAdapterRunner`
- composer value and command types that do not depend on React

Core types should express UI state only. They should not encode vendor protocol details or backend execution semantics.

### Transcript Layer

The `uinify/react` subpath owns default rendering and renderer override seams.

Stable transcript responsibilities:

- `ChatRoot`
- `MessageList`
- `Message`
- `MessagePart`
- `StepBlock`
- `ImagePart`
- `ReasoningBlock`
- `ArtifactContainer`
- renderer overrides such as `renderStep`, `renderImage`, `renderReasoning`, and artifact registry hooks

Default UI should be usable without customization, but consumers must be able to replace step, image, reasoning, and artifact rendering without replacing the entire transcript.

### Composer Layer

The `uinify/composer/lexical` subpath owns text input, attachments, and command selection.

Stable composer responsibilities:

- `LexicalComposer`
- unified command props
- attachment props
- submit value types
- command selection payloads

Slash and mention behavior should become trigger mechanisms for one unified command model. They should not remain separate long-term data models.

### Adapter Layer

The adapter layer continues to help consumers map host protocols into canonical UI events.

Adapters should terminate at `UiStreamEvent[]` plus diagnostics. They should not become vendor SDKs or official compatibility packages.

### Theme, Docs, And Examples Layer

The theme layer should continue to use CSS tokens, stable `data-slot` regions, and `slotClassNames`.

Docs and examples should explain the layered API and include one v0.4 showcase that combines commands, images, and agent steps.

## Sub-Projects

### P0: Canonical Agent Primitives

Define the data model and event layer before expanding UI or composer behavior.

Deliverables:

- Add `UiStepPart`.
- Add step stream events.
- Remove the separate `tool-call` model from the canonical recommended surface.
- Add image stream events.
- Define unified composer command value types.

Success criteria:

- Runtime can apply step and image events.
- Tool calls are represented as `step` parts with `category: "tool"`.
- Docs explain the relationship between steps, tools, images, and commands.

### P0: Transcript Agent UI

Make agent execution readable by default and customizable by layer.

Deliverables:

- Add default `StepBlock` rendering.
- Render step status and category clearly.
- Support image rendering through canonical image parts.
- Add renderer override support for steps and images.
- Add stable step and image slots.

Success criteria:

- `MessageList` can render mixed reasoning, steps, images, artifacts, attachments, and text.
- Consumers can override step and image rendering without replacing `MessageList`.

### P0: Composer Command Primitives

Upgrade slash and mention behavior into a unified command contract.

Deliverables:

- Add a unified `UiComposerCommand` model.
- Support command fields such as `id`, `kind`, `label`, `description`, `insertText`, `group`, `disabledReason`, and `metadata`.
- Submit selected commands in structured payloads.
- Use slash and mention as command triggers over the same command list.

Success criteria:

- A consumer can model SubAgents, MCP tools, skills, mentions, and custom slash actions with one command shape.
- Submit payloads include selected command metadata and do not rely only on parsing final text.

### P1: Multimodal Composer-To-Transcript Flow

Create a minimal image path from composer to transcript.

Deliverables:

- Allow image attachments to become user message image or attachment parts on submit.
- Clarify host responsibility for upload, remote URLs, object URLs, validation, and cancellation.
- Add a copyable docs template for sending image input.

Success criteria:

- Consumers can implement paste or upload image, submit, and render image in transcript using documented public contracts.

### P1: Layered Docs And Showcase

Turn v0.4 primitives into a coherent adoption path.

Deliverables:

- Add a layered public API guide.
- Add agent steps documentation.
- Add composer commands documentation.
- Add multimodal image documentation.
- Update stability guidance.
- Add a playground showcase flow that combines command selection, image input, agent steps, and final output.

Success criteria:

- New consumers can identify which layer to use for their integration.
- The showcase proves that commands, images, and agent steps compose through public contracts.

## Core Data Model

### Step Parts

`UiStepPart` should become the unified representation for agent execution nodes.

Recommended shape:

```ts
type UiStepPart = {
  id: string
  kind: "step"
  category: "tool" | "retrieval" | "handoff" | "planner" | "workflow" | "skill" | "custom"
  status: "pending" | "running" | "complete" | "error"
  label: string
  summary?: string
  inputSummary?: string
  outputSummary?: string
  error?: string
  startedAt?: string
  completedAt?: string
  metadata?: Record<string, unknown>
}
```

Tool calls are represented as `UiStepPart` with `category: "tool"`.

There should not be a separate long-term `tool-call` message part in v0.4. Existing examples, adapters, tests, and docs should migrate to step events and step parts.

### Step Events

Recommended event names:

- `part.step.started`
- `part.step.updated`
- `part.step.completed`
- `part.step.failed`

These events update one step part by `messageId` and `partId`. They should allow partial updates so host protocols can map incremental agent progress without encoding backend-specific event shapes.

### Image Parts

`UiImagePart` should remain the canonical image message part and gain enough metadata for default rendering.

Recommended shape:

```ts
type UiImagePart = {
  id: string
  kind: "image"
  url: string
  alt?: string
  mimeType?: string
  width?: number
  height?: number
  sourceAttachmentId?: string
}
```

Recommended event:

- `part.image.emitted`

The event payload should carry image metadata only. `partId` supplies the canonical image part `id`, and the runtime supplies `kind: "image"` when it writes the `UiImagePart` into message state.

Images should not use delta events. Upload progress remains attachment lifecycle state, not image state.

### Composer Commands

Commands should use one shape regardless of trigger type.

Recommended shape:

```ts
type UiComposerCommandTrigger = "/" | "@"

type UiComposerCommand = {
  id: string
  kind: "slash" | "mention" | "agent" | "tool" | "skill" | "mcp" | "custom"
  label: string
  insertText: string
  trigger?: UiComposerCommandTrigger
  description?: string
  group?: string
  disabledReason?: string
  metadata?: Record<string, unknown>
}
```

Submit values should include selected command records, not just parsed text.

Recommended selection shape:

```ts
type UiComposerCommandSelection = {
  id: string
  kind: UiComposerCommand["kind"]
  label: string
  insertText: string
  trigger: UiComposerCommandTrigger
  range: { start: number; end: number }
  description?: string
  group?: string
  metadata?: Record<string, unknown>
}
```

## Data Flow

### Transcript Flow

Host or backend events flow through the same pipeline as previous versions:

1. Host protocol emits an event.
2. Consumer mapper converts it into `UiStreamEvent[]`.
3. Consumer dispatches events into `createChatRuntime`.
4. Runtime applies canonical state transitions.
5. React transcript components render message parts.

### Composer Flow

Composer flow remains host-controlled:

1. User types text, selects commands, and adds image attachments.
2. `LexicalComposer` submits text, attachments, and selected commands.
3. Host decides whether to upload, reject, transform, or dispatch user message events.
4. Host dispatches text, image, and attachment events into the runtime.
5. Assistant events arrive later through the host's normal stream mapping path.

`uinify` should not automatically upload files, call a model, execute commands, or invoke tools.

## Error Handling

### Step Errors

Step errors are UI state, not runtime exceptions.

Rules:

- `status: "error"` marks a failed step.
- `error` stores a user-visible summary.
- `outputSummary` may preserve partial output.
- `message.failed` is only used when the entire assistant message fails.

This supports both recoverable tool failures and full assistant failures.

### Image Errors

Image errors split into two layers:

- Selection, validation, and upload errors use attachment rejection and attachment status.
- Render failures use an `ImagePart` fallback and do not mutate runtime state.

`part.image.emitted` should represent an image URL that the host believes is displayable.

### Command Errors

Commands are input intent, not execution results.

Rules:

- `disabledReason` prevents selection.
- Host code decides what to do if a submitted command is no longer valid.
- `uinify` does not validate whether MCP tools, skills, or agents exist.

### Adapter Diagnostics

Adapter validation should extend to v0.4 events.

Diagnostics should cover:

- step events before `message.started`
- step completion or failure without a prior step start/update when that can be detected
- image events without a URL

Diagnostics remain warning-first and should not block runtime dispatch by default.

## Breaking Migration

v0.4 should prefer a clean model over long compatibility.

Breaking changes:

- Remove `tool-call` from the canonical `UiMessagePart` model.
- Remove `part.tool.updated` from the recommended `UiStreamEvent` model.
- Replace `ToolCallBlock` in default rendering paths with `StepBlock`.
- Migrate examples and reference adapters from tool events to step events.
- Update docs to describe tools as `step` parts with `category: "tool"`.

If any legacy files remain temporarily during implementation, they should not be documented as v0.4 public API.

## Testing Strategy

### Core Runtime Tests

Add coverage for:

- step start, update, complete, and fail events
- `category: "tool"` replacing previous tool-call scenarios
- image emitted events
- missing message warnings or adapter diagnostics
- exhaustive `UiMessagePart` rendering after tool-call removal

### React Tests

Add coverage for:

- mixed messages with text, reasoning, step, image, artifact, and attachment parts
- `StepBlock` rendering by status and category
- step renderer overrides
- image renderer fallback
- stable step and image slots

### Composer Tests

Add coverage for:

- unified command rendering and selection
- slash and mention triggers over the unified command list
- disabled command behavior
- structured command selections in submit payloads
- image attachment submit flow producing host-usable payloads

### Docs And Example Tests

Add coverage for:

- showcase rendering command, image, and step flow
- docs path and template metadata alignment
- reference adapters producing step events instead of tool events

## Showcase Flow

The v0.4 playground should add one integrated showcase.

Flow:

1. User selects an agent command such as `/agent research` or `@researcher`.
2. User attaches an image.
3. Host maps submit payload into a user message with text, command metadata, and image part.
4. Assistant message renders reasoning.
5. Assistant message renders planner, retrieval, tool, and skill steps.
6. Assistant message renders final text or artifact.

The showcase should not call a real backend. It proves that public contracts compose.

## Release Criteria

v0.4 is ready when:

- Tool calls are represented through `UiStepPart` with `category: "tool"`.
- The legacy tool-call model is removed from recommended public docs and examples.
- Step, image, and command primitives are documented as layered public APIs.
- Default transcript UI can render agent steps and images.
- Composer can submit structured command selections and image attachments.
- The example playground includes the integrated showcase flow.
- `pnpm typecheck`, `pnpm test`, `pnpm build`, and `pnpm docs:build` pass.

## Risks

- Removing tool-call compatibility touches model types, runtime events, React rendering, examples, docs, and tests.
- The current composer implementation is large; command work should avoid making `lexical-composer.tsx` more monolithic.
- Step fields can grow into an observability schema if not kept lightweight.
- Docs must clearly explain that commands are input intent while steps are transcript output.

## Recommended Implementation Order

1. Canonical Agent Primitives
2. Transcript Agent UI
3. Composer Command Primitives
4. Multimodal Composer-To-Transcript Flow
5. Layered Docs And Showcase
