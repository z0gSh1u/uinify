# Uinify v0.3 Message Actions Design

## Summary

`v0.3` should introduce a stable message and part interaction contract.

The goal is not to hard-code product logic for every possible transcript action. The goal is to provide a consistent model, callback surface, and stable UI extension points for common transcript actions such as copy, retry, regenerate, reasoning toggles, tool detail toggles, and artifact view actions.

## Goal

After this workstream, transcript interactions should feel like a first-class product surface rather than an accumulation of isolated custom buttons.

## Scope

This workstream includes:

- message-level action contract
- part-level action contract
- visibility and availability rules for common actions
- host callback surface for action execution
- stable slots or hook points for rendering action areas
- default UI affordances for a minimal high-value action set

## Recommended Action Set

The initial stable action set should cover:

- copy
- retry
- regenerate
- toggle reasoning
- open or switch artifact views
- toggle tool details

This is enough to make the transcript feel product-grade without turning the UI into an overbuilt toolbar framework.

## Non-Goals

This workstream does not include:

- server orchestration logic
- branching conversation model
- workflow automation engine
- application-specific action suites

## Architecture Direction

`uinify` should own action presentation and action payload structure.

The host should own the business behavior behind actions such as retry or regenerate.

The intended flow is:

message or part state -> available actions -> user trigger -> structured action payload -> host handler

This keeps `uinify` focused on UI and interaction contracts rather than becoming an agent runtime.

## Contract Direction

The stable contract should define:

- action identifiers
- structured action payload shape
- which action targets are message-level versus part-level
- state-based availability rules

For example, artifact actions should attach enough artifact context for the host to understand what the user acted on, while retry and regenerate should attach enough message context for the host to decide how to execute the request.

## UI Direction

The UI should provide:

- clear action locations in the transcript
- stable slots around action areas
- consistent behavior across default and customized rendering

The library should offer useful default affordances without preventing hosts from replacing or extending the action presentation.

## Testing Strategy

Testing should include:

- action availability by message or part state
- host callback invocation with stable payload shapes
- slot and override stability around action regions
- coexistence with reasoning, tool-call, and artifact rendering

## Success Criteria

This workstream is successful when:

- transcript actions are no longer ad hoc extensions
- hosts can wire business behavior through structured action callbacks
- default transcript interactions feel more product-grade
- action UI can be customized without breaking stable slot semantics

## Recommendation

Keep the initial scope narrow and high value.

This workstream should make transcript interactions coherent and dependable without turning `uinify` into a workflow engine.
