# Uinify v0.4 Multimodal Composer-To-Transcript Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Provide a documented image path from composer attachments to transcript image parts without adding upload transport.

**Architecture:** Keep uploads host-owned. Ensure image files preserve enough metadata in composer submit values, then add a copyable example template that maps an image attachment into `part.image.emitted` for a user message.

**Tech Stack:** React, TypeScript, Lexical attachment plugin, Testing Library, Vitest, existing example templates.

---

## File Structure

- Modify: `src/composer/contracts.ts` - ensure `UiComposerAttachment` preserves image-capable submit metadata: `file`, `name`, `mimeType`, `size`, `status`, and stable attachment identity.
- Modify: `src/composer/lexical/plugins/attachment-plugin.tsx` - preserve pasted and dropped image file metadata without starting upload transport.
- Modify: `src/composer/lexical/plugins/attachment-plugin.test.tsx` - add TDD coverage for pasted/dropped image metadata and non-image attachment behavior.
- Modify: `src/composer/lexical/lexical-composer.tsx` - include image attachment data in submit values and leave upload orchestration to the host.
- Modify: `src/composer/lexical/lexical-composer.test.tsx` - add submit-value coverage for image attachments and unchanged non-image attachments.
- Create: `examples/templates/multimodal-template.tsx` - copyable template that maps a submitted image attachment into `message.started`, `part.image.emitted`, and `message.completed`.
- Modify: `examples/templates/upload-template.tsx` - only if needed to keep the existing upload lifecycle template focused on attachment orchestration, not transcript image mapping.
- Modify: `examples/fixtures/index.ts` - register the multimodal template if it is included in the playground template list.
- Modify: `examples/tests/templates.test.tsx` - assert the multimodal template renders a user image part after submit or seeded submit flow.

## Boundaries

- [ ] Do not add backend upload services, upload transport, automatic upload starts, remote URL negotiation, or vendor SDK integration.
- [ ] Do not add a rich file preview registry for arbitrary file types.
- [ ] Keep image handling minimal: preserve file metadata in composer values and show how host code can emit canonical image events.
- [ ] Keep non-image attachment behavior unchanged except where tests prove metadata preservation must be generalized.
- [ ] Use the canonical image event from the primitives plan: `part.image.emitted`.
- [ ] Preserve host ownership of validation, object URL creation, remote upload, cancellation, retry, and cleanup.

```ts

  name: string
  mimeType: string
  size: number
```

Use the existing status union rather than introducing a new lifecycle. Newly pasted and dropped files should use the existing `"queued"` status.

- [ ] Ensure dropped images use the same attachment path as other dropped files.

Do not add a separate image upload branch. The output should be normal composer attachment data with image MIME metadata preserved.

- [ ] Ensure no upload starts automatically.

The plugin may call existing `onAttachmentsChange` or equivalent local-state callbacks, but must not call network APIs, create remote URLs, or invoke host upload side effects on paste/drop.

```ts
const imageAttachment = value.attachments.find((attachment) =>
  attachment.mimeType.startsWith("image/"),

```

Then map the image attachment into canonical stream events:

```ts
runtime.appendEvents([
  { type: "message.started", messageId, role: "user" },
  {
    type: "part.image.emitted",
    messageId,
    partId,
    image: {
      url,
      alt: imageAttachment.name,
      mimeType: imageAttachment.mimeType,
      sourceAttachmentId: imageAttachment.id,
    },
  },
  { type: "message.completed", messageId },
])
```

Use the actual runtime append API name from the codebase. If the runtime expects events one at a time, adapt the example to the existing API while preserving the event order.

- [ ] Keep upload transport outside the template.

- [ ] Clarify object URL cleanup if object URLs are created.

If `URL.createObjectURL` is used, store created URLs and call `URL.revokeObjectURL` when messages are removed or when the component unmounts. Include concise code so copy-paste consumers do not leak object URLs.

- [ ] Handle non-image attachments conservatively.

If no image attachment exists, the template should either submit text normally or show a local validation message. It must not attempt to render non-image files through `part.image.emitted`.

- [ ] Avoid a rich preview registry.
