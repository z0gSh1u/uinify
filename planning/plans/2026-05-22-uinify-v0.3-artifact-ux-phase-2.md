# Uinify v0.3 Artifact UX Phase 2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make artifact rendering genuinely useful by default for `code`, `text`, and `json` artifacts, with clearer metadata, better view UX, and resilient empty or failure fallbacks.

**Architecture:** Build on the current `ArtifactContainer` and `ArtifactBody` shells instead of adding a second artifact system. Improve selection logic, empty-state handling, metadata presentation, and default renderers while preserving the existing registry and fallback extension model.

**Tech Stack:** TypeScript, React, Vitest, Testing Library, existing `ArtifactContainer`, `ArtifactBody`, `ErrorBoundary`, and CSS token system

`src/react/artifact-body.tsx`

```tsx
function getViewContent(view: UiArtifactView): string {
  if (typeof view.value === "string") {
    return view.value
  }

  return JSON.stringify(view.value, null, 2)
}

function renderBuiltInArtifactBody({ artifact, view }: ArtifactRendererProps) {
  const content = getViewContent(view)

  switch (artifact.kind) {
    case "code":
      return (
        <pre data-slot="artifact-default-code">
          <code>{content}</code>
        </pre>
      )
    case "json":
      return <pre data-slot="artifact-default-json">{content}</pre>
    case "text":
      return <div data-slot="artifact-default-text">{content}</div>
    default:
      return renderDefaultArtifactBody({ artifact, part: { id: "default", kind: "artifact", artifact }, view })
  }
}

export function ArtifactBody({ artifact, part, view }: ArtifactBodyProps) {
  // existing registry / fallback logic first
  return renderBuiltInArtifactBody({ artifact, part, view })
}
```

## Built-In Defaults

Without any registry overrides, `uinify` now provides stronger defaults for three artifact kinds:

- `code`: rendered with a code-oriented body
- `text`: rendered as readable plain content
- `json`: rendered as formatted structured content

Unknown artifact kinds still stay readable through the shared fallback path. If an artifact has no views, the container shows a stable empty-state message instead of rendering nothing.
