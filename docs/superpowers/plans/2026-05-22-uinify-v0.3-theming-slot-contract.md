# Uinify v0.3 Theming And Slot Contract Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Clarify and stabilize the slot and styling contract so consumers can integrate `uinify` into real design systems without depending on brittle DOM accidents.

**Architecture:** Do not redesign the visual system. Standardize the existing `data-slot`, `data-state`, and `slotClassNames` contract across transcript, artifact, composer, attachment, and action surfaces, extend token coverage only where major surfaces are underspecified, and document which extension path to choose for styling versus renderer replacement.

**Tech Stack:** TypeScript, React, Vitest, Testing Library, CSS custom properties, current `ChatRoot` slot-class context and stable `data-slot` attributes

`docs/advanced/stability.md`

```md
## Stable Styling Hooks

The following hooks are safe to depend on for long-term styling:

- CSS custom properties under the `--uinify-*` namespace
- Documented `data-slot` regions such as `message`, `message-parts`, `message-actions`, `part-actions`, `artifact-container`, `artifact-tabs`, `attachment-tray`, and `attachment-part`
- `slotClassNames` keys that map directly to those stable regions

Prefer tokens first, then stable slots, and replace renderers only when the default structure is the wrong product shape rather than merely the wrong visual treatment.
```
