# Uinify v0.3 Stable Adapter API Design

## Summary

`v0.3` should stabilize the adapter layer as a formal integration contract.

`v0.2` introduced reference mapper examples, but they remain example-only patterns. `v0.3` should define a small stable contract for writing mappers that convert host protocol events into canonical `UiStreamEvent` sequences, along with lightweight diagnostics and validation helpers.

The goal is not to ship official vendor adapters. The goal is to make it safe for consumers to build and maintain their own adapters against `uinify`.

## Goal

After this workstream, consumers should be able to build host-specific protocol adapters against a documented stable surface instead of copying example code blindly.

## Product Shape

This workstream should define three layers.

### 1. Stable Core Adapter Contracts

The stable public surface should define:

- adapter function shape
- mapper output contract
- diagnostics shape for invalid or suspicious output

These contracts should remain protocol-agnostic.

### 2. Stable Helper Surface

Provide a small set of reusable utilities to support adapter authors, such as:

- event normalization helpers
- validation helpers for canonical event sequences
- diagnostics helpers for ordering or missing-message issues

These helpers must stay lightweight and must not become a vendor-specific framework.

### 3. Reference Protocol Recipes

The existing OpenAI-like, agent-like, and custom-minimal examples should remain reference implementations.

However, after `v0.3`, they should be expressed as recipes built on the stable adapter contract rather than as standalone ad hoc examples.

## Scope

This workstream includes:

- stable adapter authoring contracts
- stable diagnostics and validation contracts
- lightweight adapter helpers
- updated examples built on those stable contracts
- docs explaining how to write adapters against the stable surface

## Non-Goals

This workstream does not include:

- official vendor adapters
- stable protocol-specific packages
- transport-specific helpers for SSE or WebSocket orchestration
- backend SDK wrappers

## Architecture Direction

The adapter API should terminate at canonical `UiStreamEvent[]`.

The stable adapter layer may help validate, normalize, and emit canonical events, but it must not pull backend semantics into the runtime core.

The intended data flow is:

host protocol event -> adapter contract -> canonical `UiStreamEvent[]` -> runtime

## API Direction

The stable surface should likely expose concepts equivalent to:

- adapter context or runner input
- adapter result containing canonical events plus diagnostics
- helper functions for validating event ordering or required fields

The public API should be small enough that teams can understand it quickly and implement their own adapter logic without inheriting a complex abstraction framework.

## Diagnostics

Diagnostics are important because adapter bugs are otherwise difficult to debug.

The stable adapter layer should be able to surface issues such as:

- part events before `message.started`
- malformed artifact payloads
- unsupported event combinations
- empty or suspicious mapped output

Diagnostics should help the host developer without changing runtime semantics silently.

## Testing Strategy

Testing should include:

- adapter contract unit tests
- diagnostics tests for invalid or suspicious mapped output
- runtime integration tests proving mapped events feed `createChatRuntime` correctly
- updated recipe tests for OpenAI-like, agent-like, and custom-minimal examples

## Success Criteria

This workstream is successful when:

- the adapter authoring contract is small, documented, and stable enough to recommend publicly
- consumers no longer need to reverse-engineer example mappers just to know where stable boundaries are
- example mappers become reference implementations built on the stable contract
- diagnostics make mapping failures easier to understand during integration

## Recommendation

Treat this as the most important `v0.3` workstream.

It resolves the single biggest remaining gap between `uinify` being easy to demo and `uinify` being safe to integrate into a real product codebase.
