# Uinify v0.3 Example App Templates Design

## Summary

`v0.3` should upgrade examples from scenario demos into more copyable integration templates.

The current examples and docs already communicate much more than `v0.1`, but many teams still need clearer answers to the question: what should real host code look like?

## Goal

After this workstream, `uinify` should include example apps or templates that feel close enough to production integration shapes that teams can reasonably copy from them.

## Scope

This workstream includes:

- template-level example structures
- clearer audience segmentation
- stronger alignment between docs and examples

## Template Set

The recommended minimum template set is:

- minimal app template
- adapter integration template
- upload orchestration template
- artifact customization template

These templates should not attempt to be full products. They should be realistic integration references.

## Audience Segmentation

Examples should remain organized by user intent:

- getting started
- integration
- advanced customization

But `v0.3` should push them further toward realistic host code rather than only fixture-based display logic.

## Documentation Relationship

Docs explain the contract. Templates show the shape of real usage.

This workstream should strengthen that relationship so each major doc path has a corresponding example or template path.

## Non-Goals

This workstream does not include:

- full demo product
- session management shell
- bundled backend services
- second framework layered on top of `uinify`

## Testing Strategy

Testing should include:

- example app buildability
- scenario coverage for documented example paths
- consistency between docs and example code
- smoke coverage for template structure remaining intact

## Success Criteria

This workstream is successful when:

- application teams can copy example structures into real projects with minimal translation
- advanced users can identify deeper customization patterns from templates rather than reverse-engineering playground-only code
- docs and examples reinforce each other instead of drifting apart

## Recommendation

Treat these examples as part of the product surface.

They should be optimized for copyability and clarity, not just for demonstrating that the UI can render a few scenarios.
