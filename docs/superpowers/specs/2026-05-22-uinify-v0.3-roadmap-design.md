# Uinify v0.3 Roadmap Design

## Summary

`v0.3.0` should be a platform stabilization release for `uinify`.

`v0.2` established stronger integration primitives: richer artifact modeling, host-driven attachment lifecycle, reference mappers, and adoption-focused documentation. `v0.3` should build on that work by turning the most valuable extension seams into more dependable platform contracts while also improving the default user experience enough that `uinify` feels product-grade out of the box.

The goal is not to widen `uinify` into a full chat application framework. The goal is to make the library more trustworthy for real product teams by stabilizing its integration boundary, improving its default interaction surface, and clarifying how advanced consumers should extend it.

## Version Goal

After `v0.3`, a consumer should be able to answer six practical questions confidently:

1. Which adapter-layer contracts are stable enough to build against?
2. Which transcript actions are host-owned versus library-owned?
3. Are artifacts usable by default even without deep customization?
4. Can attachment orchestration support realistic validation, failure, and recovery behavior?
5. Which theming and slot hooks are safe to depend on long term?
6. Which example apps or templates should my team copy when integrating `uinify`?

That is the release-level success definition.

## Positioning

`v0.3` should strengthen `uinify` as a chat UI foundation, not reposition it as a full app framework.

The release should preserve the current product thesis:

- React-first
- backend-agnostic
- host-controlled orchestration
- canonical model and event layer
- opinionated defaults with extension seams

## Scope

`v0.3` includes six workstreams.

### P0

1. Stable Adapter API
2. Message Actions And Interaction Contract
3. Artifact UX Phase 2

### P1

4. Attachment Lifecycle Phase 2
5. Theming And Slot Contract Polish
6. Example Apps As Integration Templates

## Recommended Order

Implementation order should be:

1. Stable Adapter API
2. Artifact UX Phase 2
3. Message Actions And Interaction Contract
4. Attachment Lifecycle Phase 2
5. Theming And Slot Contract Polish
6. Example Apps As Integration Templates

This order prioritizes stable platform surfaces first, then product-grade defaults and integration polish.

## Release Strategy

`v0.3` should remain a single coordinated release, but it should be designed and executed as one umbrella roadmap plus multiple focused sub-project specs.

That means:

- one umbrella roadmap defines the release theme, ordering, and release criteria
- each workstream gets its own focused design spec
- implementation planning should happen per workstream rather than in one monolithic plan

This keeps the version legible without forcing every subsystem into one oversized spec or plan.

## Architecture Direction

`v0.3` should not add a new top-level architecture. It should deepen and stabilize the architecture already in place.

### Model And Event Layer

The model and event layer should remain canonical and frontend-oriented.

`v0.3` should avoid protocol-specific modeling in the core. Even when stabilizing the adapter layer, the stable library contract should still terminate at canonical `UiStreamEvent` and canonical model types.

### Runtime Layer

The runtime should remain focused on canonical state and state transitions.

`v0.3` may add more structured interaction and orchestration hooks, but it should not absorb backend transport semantics or agent runtime responsibilities.

### UI Layer

The UI layer should become more product-grade by improving:

- transcript actions
- artifact usability
- attachment usability
- theming and slot clarity

The UI layer should still depend on canonical state and host callbacks rather than backend-specific primitives.

### Docs And Example Layer

For `v0.3`, docs and examples should evolve from explanatory material into a clearer adoption system:

- docs explain the contracts
- templates show how real integrations should be structured

## Release Criteria

`v0.3.0` is ready only when all of the following are true:

- adapter-layer contracts are stable enough to document as recommended public surface
- transcript interaction behavior is defined and testable through stable action contracts
- default artifact UX supports real product usage without mandatory deep customization
- attachment orchestration supports realistic validation and recovery flows
- theming and slot guidance clearly distinguish stable hooks from implementation detail
- example apps function as realistic integration templates rather than only scenario demos
- tests, typecheck, and build all pass

## Non-Goals

`v0.3` explicitly does not include:

- official vendor adapter packages
- upload client or storage transport layer
- page shell, sidebar, session list, or app chrome
- artifact workbench or IDE-like studio
- websocket transport expansion
- branching conversation UI
- voice input

These exclusions keep the release aligned with `uinify` as a foundation library instead of drifting toward a full application framework.

## Recommendation

Proceed with `v0.3` as a platform stabilization release.

The most important shift from `v0.2` to `v0.3` is not feature count. It is confidence:

- more stable extension contracts
- more usable defaults
- clearer integration guidance

That combination is the strongest next step for `uinify`.
