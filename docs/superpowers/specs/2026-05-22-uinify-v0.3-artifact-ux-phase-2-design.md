# Uinify v0.3 Artifact UX Phase 2 Design

## Summary

`v0.3` should turn the current artifact architecture into a more product-grade default experience.

`v0.2` established a stronger artifact model, shared container, view system, and renderer registry. `v0.3` should improve the default artifact experience so consumers get a stronger out-of-the-box UX even before adding custom registry renderers.

## Goal

After this workstream, artifacts should feel meaningfully usable by default rather than merely architecturally extensible.

## Scope

This workstream includes:

- stronger default artifact kinds
- clearer view UX
- better metadata presentation
- stronger empty, unknown, unsupported, and renderer-failure fallbacks

## Default Artifact Kind Set

The minimum default set should include:

- `code`
- `text`
- `json`

An optional additional kind may be included if scope allows:

- `markdown`

The purpose is not to maximize kind count. The purpose is to make the default artifact system more genuinely useful.

## View UX

The artifact container should provide stronger default view behavior:

- clear tab labeling
- predictable default-view selection
- clearer active-view state
- better empty-state and unsupported-state messaging

## Metadata UX

Artifact metadata should render more consistently across artifact kinds.

This includes clearer rules for:

- title display
- kind labeling
- source or descriptive metadata
- stable fallback behavior when metadata is sparse

## Fallback Direction

Fallback behavior matters because registry extension is now a first-class path.

`v0.3` should ensure:

- unknown artifact kinds still render meaningfully through shared fallback behavior
- missing or partial view data degrades clearly
- renderer errors can fall back to stable artifact UI rather than making the artifact block feel fragile

## Non-Goals

This workstream does not include:

- artifact workbench
- IDE-like studio UI
- side-by-side pane system
- full document editing workflows

## Testing Strategy

Testing should include:

- default renderer coverage for supported kinds
- empty or unknown artifact fallback coverage
- renderer failure isolation
- metadata rendering consistency
- view switching behavior and default view selection

## Success Criteria

This workstream is successful when:

- default artifact rendering is useful in real product scenarios without mandatory customization
- fallback behavior remains stable and readable
- metadata and view UX are coherent across artifact kinds
- consumers still retain clear extension power through the registry system

## Recommendation

Keep this workstream focused on better defaults, not larger architecture.

The architecture is already in place. `v0.3` should make that architecture feel worth using immediately.
