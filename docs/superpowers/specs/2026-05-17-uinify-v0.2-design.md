# Uinify v0.2 Design

## Summary

`v0.2.0` is an integration-first release for `uinify`.

The goal is not to broaden the library with as many new UI features as possible. The goal is to make `uinify` substantially easier to adopt, extend, and learn in real product teams. `v0.2` should take the current `v0.1` foundation and close the biggest adoption gaps around protocol mapping, attachment lifecycle handling, artifact extensibility, and practical integration guidance.

This version includes four workstreams only:

- reference mappers and adapter examples
- attachment upload lifecycle state model
- richer artifact model and renderer architecture
- consumer docs and integration examples

Each workstream must ship as a minimal usable loop, not a maximal feature set.

## Current Context

`v0.1` now provides a credible base library:

- canonical message and part model
- normalized stream event contracts
- runtime reducer and chat runtime helpers
- SSE transport helper
- default React message rendering components
- `ChatRoot`, `MessageList`, and virtualization
- Lexical-based composer shell
- slash, mention, and attachment input plugins
- theme tokens, stable slot hooks, and renderer overrides
- initial example playground

This means `v0.2` should not restart the architecture. It should build directly on the existing layering and public API shape.

## Version Goal

`v0.2.0` should make `uinify` feel like a real integration foundation rather than just a component library.

After `v0.2`, a new consumer should be able to answer four practical questions quickly:

1. How do I map my backend stream into `UiStreamEvent`?
2. How do I manage attachment uploads without giving up backend independence?
3. How do I render artifacts beyond a single code block and extend that system later?
4. Which parts of this library are stable contracts versus reference-only patterns?

That is the primary success definition for the release.

## Scope

`v0.2` includes exactly these four areas.

### 1. Reference Mappers And Adapter Examples

Ship high-quality reference mappers and integration examples for common host protocol shapes.

### 2. Upload Lifecycle State Model

Extend attachment handling so the UI and runtime can represent realistic upload state transitions while still leaving actual network upload logic to the host application.

### 3. Artifact Model And Renderer Architecture

Upgrade artifacts from a narrow code-block concept into a small but extensible subsystem with richer renderer contracts and container behavior.

### 4. Consumer Docs And Integration Examples

Ship documentation and examples that serve both adoption-focused application teams and advanced users extending the library.

## Non-Goals

`v0.2` explicitly does not include the following:

- first-party stable protocol adapters
- official upload client or transport layer for file uploads
- artifact workbench or multi-pane artifact studio
- voice input
- page-shell UI such as session list, sidebar, or app chrome
- WebSocket transport support
- branch or tree transcript UI

The purpose of these exclusions is to keep the version tightly aligned with adoption and extensibility rather than drifting into backend coupling or product-shell expansion.

## Release Strategy

This is a single coordinated `v0.2.0` release, not a sequence of loosely related `v0.2.x` feature drops.

However, the scope within the release must remain minimal and disciplined:

- each workstream ships one usable loop
- examples are realistic, but not exhaustive
- architecture is extended where needed, not reinvented
- no workstream should expand until its core boundary is clean

## Architecture Strategy

`v0.2` should extend the current architecture rather than adding a separate system.

### Model Layer

The model layer remains the center of the product. `v0.2` should extend it in two directions:

- richer attachment lifecycle semantics
- richer artifact identity, metadata, and view semantics

The model must continue to describe frontend state, not backend protocols.

### Runtime Layer

The runtime layer should remain protocol-agnostic. It should gain enough surface area to:

- represent attachment state transitions
- accept host-driven upload updates
- represent richer artifact payloads and view definitions
- support reference mapper ergonomics without absorbing protocol semantics

### UI Layer

The UI layer should gain:

- attachment lifecycle UI states
- richer artifact container and renderer dispatch
- example-ready public APIs for customization and integration

The UI layer must continue to consume canonical state only.

### Docs And Examples Layer

For `v0.2`, documentation and examples are first-class product output. They are not secondary polish.

This release is successful only if real teams can learn the integration path from these materials.

## Public API Direction

The stable public API promise remains intentionally narrow.

`uinify` continues to officially commit to:

- canonical model types
- normalized runtime event contracts
- runtime helpers and hooks
- React UI components
- Lexical composer entrypoints
- SSE helper surface
- styling and renderer customization contracts

The following remain non-stable reference surfaces:

- adapter example mappers
- protocol-specific mapping helpers
- host integration recipes

This distinction must be explicit in both naming and documentation.

## Workstream 1: Reference Mappers And Adapter Examples

### Goal

Help consumers translate real backend protocols into `UiStreamEvent` without making `uinify` itself protocol-specific.

### Product Shape

This workstream should ship as examples and reference helpers, not as a new stable adapter package family.

Recommended shape:

- `examples/adapters/openai-like.*`
- `examples/adapters/agent-like.*`
- `examples/adapters/custom-minimal.*`

Optional supporting helpers may exist, but only as lightweight reference utilities. They must not become an implicit framework for protocol integration.

### Required Example Families

#### OpenAI-like Streaming Chat

Cover the common case of streaming message deltas and lightweight tool-like events.

#### Agent Or Workflow Event Stream

Cover a richer event source with explicit tool lifecycle and artifact emission behavior.

#### Minimal Custom Protocol

Show a small, fully generic mapping path so consumers do not conclude that `uinify` only works well with the first two protocol families.

### Design Constraints

- reference mappers are examples, not supported adapters
- mapping logic must end at `UiStreamEvent`
- examples should be realistic enough to copy and adapt
- helpers may validate or shape events, but must not encode vendor assumptions into the library core

### Success Criteria

- a consumer can identify how their stream maps to `UiStreamEvent`
- examples show realistic event translation rather than toy constants
- no backend service is required to use these examples

## Workstream 2: Upload Lifecycle State Model

### Goal

Represent the lifecycle of attachments in a host-driven, backend-agnostic way.

### Core Principle

`uinify` owns state representation and UI. The host owns actual upload execution.

That means `uinify` should not perform uploads, sign URLs, or manage storage credentials.

### Required Lifecycle States

At minimum:

- `queued`
- `uploading`
- `uploaded`
- `error`
- `removed`

Optional progress metadata should also be supported.

### Two Attachment Contexts

`v0.2` should distinguish between:

#### Pre-send Attachments

Files that exist in the composer before message submission is complete.

#### Message-bound Attachments

Files that are already associated with a sent message and should render as message content or linked message metadata.

This distinction allows hosts to support different operational strategies without changing `uinify`’s contracts.

### Expected Host Interaction Model

The host should be able to drive attachment state transitions such as:

- file added locally
- upload started
- progress updated
- upload succeeded with remote reference
- upload failed with error metadata
- attachment removed

### UI Expectations

The UI should support:

- queued state
- in-progress state with optional progress
- uploaded state with completion affordance
- error state with retry hook points
- remove behavior
- send-policy integration points for hosts that require upload completion before send

### Design Constraints

- no official upload client
- no baked-in storage semantics
- no assumptions about upload-before-send vs upload-during-send
- state machine should be flexible enough for host orchestration but not vague or underspecified

## Workstream 3: Artifact Model And Renderer Architecture

### Goal

Upgrade artifacts from a narrow code preview into a real extensibility surface.

### Direction

`v0.2` should focus on artifact architecture first, not on shipping a huge number of artifact types.

### Core Concepts

#### Artifact Identity And Metadata

Artifacts should carry enough structured information to support richer UI and future extensions:

- stable identity
- artifact kind
- title or label
- metadata
- primary payload
- optional alternate views

#### Artifact Container

The default UI should render artifacts through a shared container responsible for:

- header
- artifact kind labeling
- metadata display
- view switching where relevant
- body renderer dispatch

#### Artifact Renderer Registry

`v0.2` should introduce a more coherent artifact renderer registry so hosts can override or extend artifact handling without patching core UI logic.

This is more important than adding many artifact kinds.

#### Artifact View Contract

Artifacts may expose one or multiple views, such as:

- source
- preview
- structured view

`v0.2` only needs the view contract, not a full workbench.

### Default Artifact Coverage

The version should include only a small set of default renderers sufficient to prove the architecture works, such as:

- `code`
- `text`
- optionally one structured artifact such as `json`

### API Direction

The current `renderArtifactCode` shape is too narrow for the longer-term artifact system. `v0.2` should move toward a more coherent artifact renderer contract rather than continuing to multiply renderer names by artifact subtype.

### Success Criteria

- artifacts are modeled as an extensible system, not a special-case block
- default UI can render a small set of artifact kinds through one container model
- hosts can register or override artifact renderers without violating the main message architecture

## Workstream 4: Docs And Consumer Integration Examples

### Goal

Make adoption and extension paths legible to two audiences:

- application teams integrating `uinify`
- advanced users extending or contributing to the system

### Documentation Layers

#### Adoption Docs

These should answer practical questions such as:

- how to embed `uinify` in an existing React app
- how to map an SSE backend into `UiStreamEvent`
- how to drive attachment upload lifecycle from host code
- how to customize theme and renderers
- how to integrate artifact rendering

These docs should be task-oriented, not just API reference pages.

#### Extension Docs

These should explain:

- why the canonical model is shaped the way it is
- runtime and reducer boundaries
- artifact renderer registry extension rules
- composer contract extension points
- how and when to build reference mappers
- what is stable API versus what is example-only guidance

### Example Taxonomy

Examples should be organized by user intent.

Recommended categories:

- `getting-started`
- `integration`
- `advanced`

### Reference Quality Standard

Examples should look like real host code, not presentation demos. They should be designed for copying and adaptation.

### Success Criteria

- a new consumer can understand the adoption path quickly
- a power user can identify extension seams accurately
- examples communicate stable contracts versus reference-only patterns clearly

## Error Handling Expectations

### Reference Mappers

- invalid mapped output should surface development-time warnings or assertions
- example helpers should not silently swallow mapping failures

### Upload Lifecycle

- attachment failures should be localized to the affected attachment
- upload state transitions should be explicit and predictable

### Artifact System

- unknown artifact types should have a stable fallback path
- missing metadata should degrade gracefully rather than crash rendering

### Docs And Examples

- examples must clearly mark reference implementation code
- unstable or example-only utilities must not be presented as core stable API

## Testing Strategy

`v0.2` should raise the testing bar beyond isolated units.

### Model And Runtime Tests

Add coverage for:

- attachment lifecycle state transitions
- richer artifact view or metadata handling

### Component Tests

Add coverage for:

- attachment lifecycle UI states
- artifact container dispatch and fallback behavior

### Integration Tests

Add coverage for:

- reference mapper output feeding runtime and UI
- host-driven upload state updates reflected in composer and message surfaces

### Example Verification

Examples should verify actual integration behavior, not just presence of headings or fixture names.

## Release Criteria

`v0.2.0` is ready only when all of the following are true:

- reference mapper examples are realistic and understandable
- upload lifecycle supports a complete host-driven UI loop
- artifact rendering has a coherent container and registry model
- adoption docs and extension docs are both present and structured
- examples cover integration, upload, artifact, and customization flows
- build, typecheck, and tests all pass

## Version Success Metrics

The release is successful if:

- a new application team can understand how to integrate `uinify`
- an advanced user can understand how to extend `uinify`
- the `v0.2` extension boundaries are stable enough that `v0.3` does not require undoing this design

## Recommendation

Proceed with `v0.2.0` as a single integration-first release that improves adoption and extensibility without widening the backend or product-shell scope.

The main design principle is disciplined expansion:

- deepen the integration path
- clarify extension contracts
- keep core runtime and UI backend-agnostic
- let examples and documentation carry protocol-specific guidance

This keeps `uinify` aligned with its strongest positioning: a React chat UI foundation that is practical for real products and still structurally extensible.
